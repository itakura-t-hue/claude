1なぜ作るのか（背景・課題・目的）

💼 ビジネス解説：なぜこのツールが必要なのか

現在のチーム体制はプリセールス3-4名、ディレクター6名、コンテンツ担当5名、エンジニア3名、セールス4-6名です。1案件にプリセールス1名+ディレクター1名が基本アサイン。案件数に対して人員が足りず、対応品質が落ちることで解約が発生しています。

このツールの最大の目的は「解約の阻止」です。効率化はその手段にすぎません。クライアントが「このツールを使い続けたい」「解約したらデータが消えてもったいない」と感じるロックイン構造を設計し、さらにコンテンツ制作プロセスの効率化で炎上→解約を防ぎます。

1.6 前提条件

⚠ 開発開始前に必要な準備
  • Ahrefs Advancedプラン（$449/月）のツール専用アカウント契約が必要 — 120社の日次同期で現行アカウントのユニット上限を超過するため（詳細はセクション3.3参照）
  • 自社GCPプロジェクトの作成（Cloud Run, Cloud SQL, BigQuery, Memorystore, Cloud Storage）
  • Google Ads APIの既存アカウントでの有効化（Keyword Planner利用）
  • GA4の管理者権限への昇格依頼 — 現行の共有用Gmailアカウントの権限を「閲覧者」→「管理者」に変更してもらう（BigQueryエクスポート設定を自社で代行するため）
💼 データ管理方針：全データを自社GCPに集約

本ツールが扱う全てのデータは自社GCPプロジェクト内で完結して管理します。

  • GA4のローデータ（イベント単位の生データ）：GA4のBigQueryエクスポート機能を使い、エクスポート先を自社GCPのBigQueryに設定。ユーザー行動ログ（ページビュー、スクロール、クリック、CV等）が毎日自動で自社BQに蓄積される。設定は共有用GmailのGA4管理者権限で自社が代行するため、クライアントのGCPは不要
  • GSC/Ahrefs等の集計データ：各APIで取得 → 自社BigQueryにINSERT
  • 保守プランのロックイン：GA4ローデータ含む全データが自社管理下にあるため、解約時にはこの蓄積データへのアクセスが全て失われる。特にGA4ローデータはGA4本体では14ヶ月で消えるが、自社BQなら永久保持。これが保守プラン提案の核心
🎯 GA4ローデータ蓄積のメリット（コンサルの提案幅が大幅拡大）

GA4 Data API（集計データ）だけでは「2月のSSは40,600」止まりです。BigQueryにローデータを蓄積すると：

  • CVユーザーの行動パス分析 — どの記事→どのサービスページ→CVした？
  • コンテンツ品質の定量評価 — ページ滞在時間×スクロール深度で「ちゃんと読まれてる記事」を特定
  • 内部リンクのクリック率 — どのCTAが踏まれているか、改善すべきリンクはどれか
  • フォーム離脱分析 — CVフォームのどの項目で離脱しているか
  • デバイス×ページ別CVR差異 — モバイルだけCVRが低いページの特定
  • 時間帯・曜日別の行動パターン — 最適な記事公開タイミングの提案
  • 新規 vs リピーターの行動差異 — リピーター向けコンテンツ戦略

これにより月2案の改善提案が「なんとなく」ではなくデータドリブンで根拠付きになります。

GA4 BigQueryエクスポートの設定フロー

  1. クライアントにGA4の権限を「閲覧者」→「管理者」に変更してもらう（共有用Gmailアカウントに対して。1回のみの作業）
  2. 自社ディレクターがGA4管理画面にログインし、[管理] → [BigQueryリンク] からエクスポートを設定
  3. エクスポート先に自社GCPプロジェクトのBigQueryデータセットを指定（クライアント別のデータセット: client_{id}_ga4）
  4. エクスポート頻度: 「毎日」を選択（ストリーミングは不要、日次で十分）
  5. 設定完了後、翌日からローデータが自動で自社BQに蓄積開始

この設定はクライアントのGCPプロジェクトを一切使用しません。自社GCPのBigQueryに直接エクスポートされるため、データは完全に自社管理下に置かれます。

このツールは、6つの外部サービスから自動的にデータを取ってきて、1つの画面に表示します。イメージとしては、「6つのお店から毎朝自動で仕入れて、1つのお惣菜コーナーに並べる」ようなものです。

  • Google Search Console（GSC） → 「どんなキーワードで何位に表示されて、何回クリックされたか」
  • Google Analytics（GA4） → 「サイトに何人来て、何ページ見て、お問い合わせは何件か」
  • Ahrefs → 「他のサイトからリンクがいくつ貼られているか、競合と比べてどうか、KWの難易度はどの程度か」
  • Google Keyword Planner → 「キーワードの月間検索ボリューム（Google公式データ）」
  • WordPress → 「クライアントのサイトにどんな記事が公開されているか」
  • PageSpeed Insights → 「サイトの表示速度は速いか遅いか」

3.2 Google Analytics 4（GA4）

💼 ひとことで言うと

サイトに来た人の行動データです。「何人来たか」「どのページを見たか」「お問い合わせ（CV）は何件か」がわかります。
現在の業務での使い方：月次レポートのスライド9「チャンネル別流入数」、スライド10「オーガニックSS推移」、スライド15「CV数推移」で使っている数値です。

⚠️ 現状：GA4データはLooker Studioに接続済み

現在、GA4のデータ閲覧はLooker Studio（旧Googleデータポータル）を使用しています。本ダッシュボードではGA4 Data APIから直接データを取得する方式を採用します。これにより以下のメリットがあります：

  • データの一元管理：Looker Studioでは他のデータソース（GSC、Ahrefs、CMS）と統合できないが、本ダッシュボードではすべてを1画面で確認可能
  • 自動レポート生成：Looker StudioはPPTX出力に非対応。本ダッシュボードでは既存のPPTXフォーマットに合わせた自動生成が可能
  • アラート機能：Looker Studioにはアラート機能がないが、本ダッシュボードでは流入急減・CV異常を自動検知してSlack通知
  • リライト提案：GA4の流入データ × GSCの順位データ × Google Keyword Plannerの検索Volをクロスで分析し、リライト優先度を自動算出

なお、Looker Studioは引き続き併用可能です。本ダッシュボードの開発・安定稼働が確認できた段階で、段階的に移行する運用を推奨します。

⚠ 保守ライト vs 保守プロの違い

保守プロ（月額¥80,000）はGA4ローデータのBigQuery蓄積が継続します。ローデータはユーザー行動の生ログ（ページビュー、スクロール、クリック等）で、蓄積が止まると分析の空白期間が発生します。フルコンサルに復帰した際に「空白期間のデータがない」状態になるため、将来の復帰可能性があるクライアントには保守プロを推奨します。

保守ライト（月額¥50,000）はGA4管理者権限が取得できないクライアントや、ローデータ分析のニーズがないクライアント向けです。集計データの同期は継続するため、ダッシュボード閲覧とレポート自動生成は通常通り動作します。

4.4 データロックイン設計

💼 ビジネス解説：「解約したらもったいない」と思わせるデータ蓄積

ダッシュボードを使い続けるほど、解約した場合に失うデータの価値が大きくなります。これがロックインの本質です。

蓄積データ解約時の影響ロックイン効果
順位履歴の長期蓄積 GSCは16ヶ月で消えるが、自社BQなら永久保持 契約期間が長いほど「年次比較」のデータが貴重になる
GA4ローデータ GA4本体は14ヶ月で消える。自社BQに蓄積したローデータは永久保持 保守プロ契約中は蓄積継続。解約すると空白期間が発生し、復帰後も分析精度が落ちる
競合比較データの履歴 Ahrefsの競合分析履歴が消える 「2年前と比べて競合がどう変わったか」の分析が不可能に
施策の因果記録 「この施策を打った → この結果になった」の記録が消える ナレッジベースの過去データが全て失われる
レポートアーカイブ 過去の月次レポート全てにアクセス不可 クライアント社内で「前年のレポート見たい」と言われても対応不可に
ポータルURL クライアント社内で共有されているダッシュボードURLが無効に 社内報告の仕組みがゼロになる

5データベース設計（全テーブル定義）

💼 ビジネス解説

データベースとは、集めたデータを整理して保管する「倉庫」です。以下は「どんな棚（テーブル）を作って、何を保管するか」の設計図です。エンジニアが正確に実装するために必要な情報なので、技術的な内容が多くなっています。ビジネス視点では「こんな種類のデータを管理するんだな」という全体像を把握していただければ十分です。

⚠ PostgreSQL + BigQuery ハイブリッド構成：テーブル振り分け

PostgreSQL（Cloud SQL）に格納するテーブル（操作系・マスタデータ）：
User, Client, Project, ProjectAssignment, Keyword, Content, ContentKeyword, ContentVersion, Task, ChatChannel, ChatChannelMember, ChatMessage, ChatAttachment, AlertRule, AlertHistory, AuditLog, SyncLog, WriterProfile, KnowledgeBase, NotificationPreference, CwvSnapshot

BigQuery に格納するテーブル（分析系・履歴データ、日付パーティション、追記専用）：

  • rank_history — 日次キーワード順位データ（日付パーティション、project_idクラスタリング）
  • traffic_snapshot — 日次プロジェクト別トラフィックデータ
  • content_traffic_history — 日次コンテンツ別トラフィック
  • backlink_snapshot — 日次被リンクメトリクス
  • ga4_events — GA4 BigQueryエクスポートによるイベント単位の生データ（自社GCPのBQに直接エクスポート。ページビュー、スクロール、クリック、CV等の全イベント）
  • ga4_daily_summary — GA4 Data APIで取得した日次集計レポート（管理者権限未取得のクライアント用フォールバック）
🛠 BigQueryテーブル設計指針

BigQueryテーブルは追記専用（INSERT only、UPDATE/DELETE不可）です。日付パーティショニングによりクエリコストを最適化。全データは自社GCPプロジェクト内のBigQueryに集約されます。

  • GA4ローデータ: GA4のBigQueryエクスポート機能で自社BQに直接送信（GA4管理者権限で自社が設定代行。クライアントのGCP不要）
  • GSC/Ahrefs/KWP等: 自社バッチ処理がAPIで取得し自社BQにINSERT
  • GA4管理者権限が未取得のクライアントは、GA4 Data APIで集計データを取得しBQにINSERT（ga4_daily_summaryテーブル）

💼 ヘルススコアとは

「このクライアントの案件は健全か？」を0〜100点で自動評価する仕組みです。点数が低いクライアントは解約リスクが高い可能性があるので、先手を打って対策できます。点数は以下の4つの要素から計算します：

  • オーガニック流入が伸びているか（30点満点）
  • KW順位が改善しているか（25点満点）
  • CV数が増えているか（25点満点）
  • 施策タスクが順調に進んでいるか（20点満点）

💼 ビジネス解説：なぜチャットが必要か

現在、クライアントとの連絡はメール、社内の連絡はSlackやLINEなど、やり取りがバラバラに分散しています。このチャット機能では：

  • 案件ごとにスレッドが分かれるので、過去のやり取りを遡りやすい
  • 「この記事の順位が落ちたので対応してください」と書くとき、実際の記事データや順位データを添付（メンション）できるので、認識の齟齬がなくなる
  • クライアントも参加できるため、メールでのやり取りが不要になり、レスポンスが速くなる
  • 全やり取りがダッシュボードに残るので、担当者変更時の引き継ぎもスムーズ

「バッチ処理」とは、毎日決まった時間に自動実行される裏方の仕事です。深夜3時に自動でGoogleやAhrefsからデータを取ってきて、データベースに保存します。翌朝ダッシュボードを開くと、前日までの最新データが表示されている状態になります。

🛠 BigQuery読み取り（ダッシュボード表示）

ダッシュボードのグラフ・チャート表示はBigQueryクライアントライブラリ（@google-cloud/bigquery）を使用してBigQueryから直接読み取ります。日付パーティションにより、特定期間のクエリは関連パーティションのみスキャンし、コストと速度を最適化します。

13技術構成・インフラ

💼 ビジネス解説

ツールを動かすための「道具」と「場所」の選定です。わかりやすく言うと：

  • Next.js = ツールの見た目と中身を作る道具（画面もデータ処理もこれ1つで作れる）
  • PostgreSQL = 日常データを保存する倉庫（ユーザー情報、タスク、チャット等の操作系データ）
  • BigQuery = 大量の分析データを保存・集計する巨大倉庫（順位履歴、トラフィック履歴等の時系列データ）
  • Redis = よく使うデータを手元に置いておく棚（表示を高速化）
  • GCP（Google Cloud Platform） = ツールを動かすクラウド基盤（自動でスケールする）
💼 なぜPostgreSQLとBigQueryの2つを使うのか

PostgreSQLは日常の操作（ログイン、タスク更新、チャット）を高速に処理します。BigQueryは重い分析処理（120社以上 x 100キーワード x 365日 = 年間438万件の順位レコード）を担当します。PostgreSQLだけでは2-3年後にデータ量が増えて動作が遅くなりますが、BigQueryなら数十億行でも数秒で集計できます。

本システムには 2つの異なるOAuth用途 が存在する。仕様上「OAuth 2.0」と一括りにされているが、実装は全く別物。

用途目的アカウントスコープ

決定が必要な事項:

  • (B) クライアントごとに別のGoogleアカウントで認証する方式 → Projectモデルのトークンフィールドをそのまま使用
  • (C) GCPサービスアカウント + ドメイン全体の委任方式 → OAuthトークン管理が不要になるが、クライアント側の設定負荷が増える

補足: 現行運用は「共有Gmail 1アカウント」でGSC/GA4に閲覧者権限を付与してもらう方式（セクション1.6参照）。この場合は (A) が整合するが、リフレッシュトークン失効時の再認証フロー（管理者が手動で共有Gmailで再ログイン）を設計に含める必要がある。

(C) Compute Engine（小型VM常駐） 自由度が高い インフラ管理コスト増

推奨: (A) がシンプル。Cloud Schedulerで定刻にHTTPリクエストを送信 → Cloud RunのAPIがBullMQにジョブを投入 → 同一インスタンス内のワーカーが処理。

P1-2. バッチジョブの部分失敗ハンドリング

以下のケースに対する挙動を定義する必要がある:

ケース現状の仕様決定が必要な事項
120社中50社目でAPIレートリミット 「次バッチに延期」 残り70社は翌日に持ち越し？即時リトライ待ち？優先度はどうなる？
1社の同期が途中失敗（GSCは成功、GA4が失敗） SyncLog partial 次回実行時に差分取得？全件再取得？ partial状態のデータは表示に使うか？
リトライ3回全失敗 記載なし 管理者通知のみ？手動再実行ボタン？自動的に翌日キューに入れる？
バッチジョブ間の依存（rewrite-scoreがsync-gsc完了前に実行） 時刻ベース イベントドリブン（前ジョブ完了をトリガー）に変更すべきか？

冪等性の設計:
全バッチジョブはべき等（何度実行しても同じ結果）であること。@@unique([keywordId, date]) 等のユニーク制約でUPSERTを使用する。

P1-3. GA4 APIクォータの検証

GA4 Data APIの制約:

  • プロパティあたり: 同時10リクエスト、日10,000トークン
  • トークン消費量はクエリの複雑さに依存（ディメンション数、日付範囲等）

検証が必要:

  • 1社あたり5レポート（R1-R5）のbatchRunReportsで消費するトークン数の実測
  • 120社で日次トークン上限（10,000 × 120 = 1,200,000トークン？ or プロジェクト全体で共有？）
  • GCPプロジェクトレベルのグローバルクォータの確認

推奨: Phase 2開始前にパイロット5社でトークン消費量を実測すること。

プロンプト設計（要プロトタイプ）:

  • 1社あたりの出力トークン数（推定 3,000-5,000トークン）

  • 不適切な出力時: テンプレートベースのフォールバック所感を表示（「前月比でオーガニックSSが{diff}%{増加/減少}しました。」等）

P3: 全Phase横断の技術課題
P3-1. マルチテナント分離の実装パターン

Prismaミドルウェアの制限:

  • $queryRaw / $executeRaw には適用されない
  • BigQueryクエリには一切効かない
  • User テーブル等、projectId を持たないテーブルには適用不可

推奨設計:

レイヤー分離方式
PostgreSQL（Prisma経由）Prismaミドルウェア + @@index([projectId])
PostgreSQL（生SQL）使用禁止。全クエリをPrisma経由に限定
BigQueryクエリに必ず WHERE project_id = @projectId を含める。共通ユーティリティ関数で強制
API層全エンドポイントで requireProjectAccess(userId, projectId) ミドルウェアを適用

テスト戦略:

  • E2Eテストで「ユーザーAがプロジェクトBのデータにアクセスできないこと」を全主要エンドポイントで検証
  • テストシードに2テナント分のデータを投入し、クロステナントアクセスの防止を網羅的にテスト

20リスクと対策

#リスク影響度発生確率対策
1 Ahrefs APIユニット上限超過 中 低（専用アカウントで軽減済み） 専用アカウント契約により、現行業務への影響はゼロ。ユニット消費量は週次取得最適化により上限の約48%で運用可能。80%到達時にSlackアラート。/api/settingsでユニット消費量のダッシュボード表示
2 クライアントCMSがWordPress以外 中 中 Phase 1はWP限定と明示。CmsConnectorインターフェースでプラグイン設計し、他CMSは後から追加可能に
3 GA4/GSCの認証トークン期限切れ 中 高 リフレッシュトークンで自動更新。失敗時はSyncLogにエラー記録 + 管理者通知。設定画面で再認証ボタン
4 データ量増加によるDB負荷 低（BigQuery導入で軽減） 低 履歴データはBigQueryに格納（日付パーティション）。BigQueryは数十億行でも数秒で集計可能。PostgreSQLは操作系データのみで負荷を抑制
5 クライアントデータの漏洩 高 低 テナント分離（Prismaミドルウェア）、暗号化保存、監査ログ、セキュリティレビュー実施
6 開発リソース不足 高 中 Phase 1のMVP機能を厳選。外部エンジニア委託の検討。shadcn/uiなど既製コンポーネントの最大活用
7 Google API仕様変更 中 低 API連携レイヤーを抽象化（DataSourceConnectorインターフェース）し、変更時の影響範囲を局所化
8 ユーザー定着しない 中 中 Phase 1リリース直後に社内トレーニング実施。2週間のパイロット運用でフィードバック収集。UIの段階的改善
9 クライアントがChatworkから移行しない 中 高 段階的移行: 新規クライアントからダッシュボードチャットを開始。既存クライアントにはポータルの便利さを体験してもらい、自発的な移行を促す。Chatwork API連携で自動転送を実装し、移行のハードルを下げる
10 外部ライターポータルからのデータ漏洩 高 低 WRITERロールには厳格な権限分離を適用。PrismaミドルウェアでClient/Projectテーブルへのアクセスを完全ブロック。ライターには記事ID以外のプロジェクト情報を一切渡さない設計
11 4フェーズによるスコープクリープ 中 中 各フェーズの厳密なMVP定義。2週間スプリントレビューで範囲の肥大化を防止。Phase間で必ずレトロスペクティブを実施し、次フェーズの優先度を再評価
