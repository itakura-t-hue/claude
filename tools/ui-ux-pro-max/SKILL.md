# UI/UX Pro Max — Design Intelligence

プロフェッショナルなUI/UXデザイン判断をサポートする包括的ナレッジベース。
コード生成時のデザインガイドとしても、デザインコンサルとしても機能する。

## ワークフロー

UIに関するタスクを受けたら、以下の4ステップで進める。

### Step 1: 要件分析

ユーザーのリクエストから以下を抽出する:

- **プロダクトタイプ**: SaaS / EC / ダッシュボード / ポートフォリオ / ブログ / LP etc.
- **ターゲットオーディエンス**: B2B / B2C / 開発者 / 一般消費者 etc.
- **スタイルキーワード**: ミニマル / グラス / ダーク / ポップ / プロフェッショナル etc.
- **テックスタック**: React / Next.js / Vue / Svelte / HTML / React Native etc.
- **特別な要件**: アクセシビリティ / パフォーマンス / ダークモード / レスポンシブ etc.

### Step 1.5: AIっぽさ排除チェック（重要）

UIを生成する前に `references/anti-ai-patterns.md` を必ず読むこと。AIが生成するUIには特徴的なクセがあり、意識しないと「あ、これAIが作ったな」とすぐバレる。主な落とし穴:

- **テキストが定型文**: 「一気通貫で」「体系的に」「包括的な」のようなAI特有のフレーズを多用してしまう → 口語的で具体的な表現に書き換える
- **レイアウトが均一すぎる**: 3カラム均等グリッドが延々と続く → カードサイズを混ぜる、非対称レイアウトを入れる、余白にリズムをつける
- **ビジュアルがない**: テキストとSVGアイコンだけ → Unsplash写真、スクリーンショット、イラストを積極的に使う
- **どのセクションも同じ構造**: 見出し→説明→3カードの繰り返し → セクションごとに構成を変える（全幅画像、2カラム+テキスト、ベントグリッド等）
- **色使いが教科書的**: プライマリ・セカンダリ・アクセントを機械的に適用 → 実際のブランドが使うような「くずし」を入れる

このステップを飛ばすと、技術的には正しいがどこかテンプレ感のあるUIになってしまう。

### Step 2: デザインシステム構築

`references/` フォルダから関連するリファレンスを読み込み、以下を決定する:

| 決定事項 | 参照ファイル | 読むタイミング |
|---------|------------|-------------|
| AIっぽさ排除 | `references/anti-ai-patterns.md` | **毎回（最優先）** |
| 全体スタイル方針 | `references/styles.md` | 毎回 |
| カラーパレット | `references/colors.md` | 毎回 |
| タイポグラフィ | `references/typography.md` | 毎回 |
| UI推論ルール | `references/ui-reasoning.md` | プロダクトタイプが明確な時 |
| UXガイドライン | `references/ux-guidelines.md` | インタラクション設計時 |
| アイコン選定 | `references/icons.md` | コンポーネント実装時 |
| チャート選定 | `references/charts.md` | データ可視化が必要な時 |
| LP設計パターン | `references/landing.md` | ランディングページ作成時 |
| Reactパフォーマンス | `references/react-performance.md` | React/Next.js実装時 |
| モバイルUI | `references/app-interface.md` | モバイルアプリ・RN・Flutter時 |
| Google Fonts一覧 | `references/google-fonts.md` | フォント選定の詳細が必要な時 |

**重要**: 全ファイルを一度に読む必要はない。タスクに関連するファイルだけを読む。典型的なWebサイト作成なら `styles.md` + `colors.md` + `typography.md` の3つで十分なことが多い。

### Step 3: デザイン判断の適用

リファレンスから得た情報を元に、具体的なデザイン判断を行う:

1. **スタイル選定**: プロダクトタイプとオーディエンスに基づいて1-2つのメインスタイルを選ぶ
2. **カラーパレット構築**: プライマリ / セカンダリ / アクセント / ニュートラルの4軸で定義
3. **タイポグラフィ設定**: 見出し用 + 本文用のフォントペアリングを選定
4. **スペーシングシステム**: 4px or 8pxベースのリズムを設定
5. **エフェクト**: シャドウ / グラデーション / アニメーションの方針を決定

### Step 4: 実装

コード生成時は以下のプロフェッショナルルールを遵守する:

## プロフェッショナルUIルール

これらは「プロが作ったUI」と「素人っぽいUI」を分ける重要なルール。

### アイコン
- 絵文字をアイコンの代わりに使わない。Lucide、Phosphor、Heroicons等のベクターアイコンライブラリを使う
- アイコンファミリーは1プロジェクト内で統一する（Lucideなら全部Lucide）
- タッチターゲットは最低44x44pt確保する
- アイコンにはaria-labelを付与する

### インタラクション
- タップ/クリックフィードバックは80-150ms以内に表示する
- トランジションは150-300ms（これ以上長いとモッサリ感じる）
- `prefers-reduced-motion` を尊重する
- ホバー状態、フォーカス状態、アクティブ状態、ディセーブル状態を全て定義する
- ローディング状態を必ず用意する（スケルトンスクリーン推奨）

### カラーとコントラスト
- テキストのコントラスト比は最低4.5:1（WCAG AA）
- 大きいテキスト（18px+ bold or 24px+）は3:1でOK
- カラーだけで情報を伝えない（アイコンやテキストを併用）
- ダークモードではコントラストを別途テストする
- セマンティックカラートークンを使う（`--color-primary` ではなく `--color-action`）

### レイアウト
- Safe Area を尊重する（ノッチ、ジェスチャーバー）
- 8dp スペーシングリズムを基本にする
- テキストの最大幅は65-75文字（読みやすさのため）
- コンテンツのジャンプ（CLS）を防ぐ（画像にwidth/heightを指定、フォント読み込み時のFOUT対策）

### レスポンシブ
- モバイルファースト設計
- ブレークポイント: sm(640px), md(768px), lg(1024px), xl(1280px), 2xl(1536px)
- タッチデバイスではホバーに依存しない
- フォントサイズは `clamp()` で流動的に

### ダークモード
- ライト/ダーク両方でコントラストをテストする
- 純粋な黒(#000)は避ける → `#0a0a0a` ~ `#1a1a1a` を使う
- エレベーション（影の代わり）は明るさで表現する
- 画像・イラストもダークモード対応を考慮する

### アクセシビリティ
- セマンティックHTML（`<nav>`, `<main>`, `<article>` 等）を使う
- キーボードナビゲーションを確保する（Tab順序、フォーカスリング）
- 見出し階層を正しく使う（h1 → h2 → h3、レベルを飛ばさない）
- フォームにはラベルを必ず付ける（プレースホルダーだけでは不可）
- エラーメッセージは具体的に（「入力が無効です」→「メールアドレスの形式が正しくありません」）

### パフォーマンス
- 画像は `next/image` や `<picture>` + WebP/AVIF で最適化
- アニメーションは `transform` と `opacity` のみ使う（`width`/`height`/`top` のアニメは避ける）
- 重いコンポーネントは遅延ロード（`React.lazy` / `dynamic`）
- フォントは `font-display: swap` + preload

## コンポーネント実装パターン

コードを書く時は、テックスタックに応じた最適なパターンを使う:

### React / Next.js
- Server Components をデフォルトにし、インタラクティブな部分だけ `"use client"`
- CSS Modules or Tailwind CSS（グローバルCSSは避ける）
- `cn()` ユーティリティ（clsx + tailwind-merge）でクラス名を結合
- shadcn/ui のコンポーネントがある場合は積極的に使用

### Vue
- Composition API + `<script setup>` を推奨
- Scoped styles を使用
- Props でデザイントークンを受け取れるようにする

### Svelte
- Svelte 5 の Runes シンタックスを使用
- CSS variables でテーマトークンを管理

### HTML / Vanilla JS
- CSS Custom Properties でデザイントークンを定義
- BEM 命名規則を推奨
- Progressive Enhancement を意識

### React Native
- `references/app-interface.md` を必ず読む
- Platform-specific なスタイルを考慮
- Safe Area を必ず処理

## Pre-Delivery チェックリスト

コードを納品する前に確認:

**AIっぽさ排除（最重要）:**
- [ ] テキストにAI定型フレーズが残っていないか（「体系的に」「包括的な」「一気通貫で」等）
- [ ] 全セクションが同じ構造（見出し→説明→3カード）の繰り返しになっていないか
- [ ] レイアウトに非対称性・リズムの変化があるか
- [ ] 写真・イラスト・スクリーンショット等のビジュアル素材が含まれているか
- [ ] カードサイズ・余白が全て均一になっていないか

**プロフェッショナル品質:**
- [ ] 絵文字をアイコンとして使っていないか
- [ ] アイコンファミリーが統一されているか
- [ ] タッチターゲットが44x44pt以上か
- [ ] コントラスト比がWCAG AA（4.5:1）を満たすか
- [ ] ホバー/フォーカス/アクティブ/ディセーブル状態が定義されているか
- [ ] ローディング状態があるか
- [ ] セマンティックHTMLを使っているか
- [ ] キーボードナビゲーションが機能するか
- [ ] `prefers-reduced-motion` を尊重しているか
- [ ] レスポンシブ対応しているか
- [ ] ダークモード対応しているか（要件にある場合）
- [ ] フォントの最適化（swap + preload）をしているか
